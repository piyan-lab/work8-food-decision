# 👩‍💼 產品需求文件 (PRD) - Food Decider 美食決策網站 (v4 雙軌解析與縣市分類版)

## 1. 提案背景與產品概述
- **產品名稱**：Food Decider (美食決策家)
- **價值主張**：透過自建的本地餐廳資料庫，搭配**雙軌地址解析機制（Google API + OSM Nominatim）**，提供穩定且支援模糊搜尋的決策體驗。新增的**縣市分類功能**更能幫助使用者妥善管理全台各地的口袋名單。

## 2. 目標與成功指標
- **業務目標**：解決免費地圖 API 對地址要求過於嚴苛的痛點，允許透過 `.env` 彈性掛載 Google API Key 來優化搜尋體驗。同時透過縣市欄位加強後台管理能力。

## 4. 功能需求 (Functional Requirements)

### 4.2. 雙重定位與智慧地址解析 (Smart Geocoding)
- **User Story**：我想要能隨意輸入「台北101」或是「逢甲夜市」，系統就能找到我的位置，不用打完整的區跟路名。
- **AC**：
  - 系統自動偵測環境變數是否有 Google Maps API Key。
  - 若有，優先使用 Google Geocoding API 解析模糊地址。
  - 若無或異常，則自動切換回開源的 Nominatim 作為備援。

### 4.5. 餐廳管理功能與縣市分類
- **User Story**：我想要在新增口袋名單時，標記它在哪個縣市，以便我未來可以快速篩選。
- **AC**：
  - 管理介面上方新增「全台 22 縣市篩選器」可過濾顯示清單，不再僅限於六都。
  - 新增/編輯表單內加入「全台 22 縣市下拉選單」。
  - 地址輸入框支援模糊搜尋（當啟動 Google API 時），大幅降低建檔門檻。

## 5. 邊界條件與例外處理 (Edge Cases)
- **Google API 額度耗盡或無效**：系統自動印出失敗 Log，並且無感降級回傳 Nominatim 的解析結果。
- **地址完全無法解析**：跳出提示「無法解析該地址，請輸入完整有效的地址。若有 Google 金鑰可支援模糊搜尋。」
